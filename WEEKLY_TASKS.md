# 工具集升级执行清单

> 本文档替代原来的按周/按天排期。  
> 项目不需要固定时间表，只需要按优先级逐步完成。  
> 每一步都应保持项目“可直接下载、可直接打开、轻量运行”。

---

## 一、执行顺序

建议按以下顺序推进：

1. 基础检查。
2. 共享工具函数完善。
3. 复制和提示体验统一。
4. 单个工具页试点。
5. 批量推广到其他工具页。
6. 移动端优化。
7. 性能敏感工具优化。
8. 新工具扩展。
9. 文档同步。

---

## 二、任务 0：基础检查

### 目标

确认当前项目状态，避免在不清楚现状的情况下重构。

### 操作

- [ ] 打开 `index.html`，确认首页可用。
- [ ] 逐个打开现有工具页，确认主要功能可用。
- [ ] 检查浏览器控制台是否有明显错误。
- [ ] 确认 `js/utils/` 中已有工具函数是否正常。
- [ ] 确认所有路径均为相对路径。

### 验收

- [ ] 首页可用。
- [ ] 现有工具页可用。
- [ ] 没有阻塞级错误。

---

## 三、任务 1：完善共享工具函数

### 目标

让常用能力可以被多个工具页复用，但不引入构建步骤。

### 文件

已存在：

- `js/utils/toast.js`
- `js/utils/copy.js`
- `js/utils/debounce.js`
- `js/utils/storage.js`

可选新增：

- `js/utils/loading.js`
- `css/themes.css`

### 操作

- [ ] 检查 `toast.js` 是否支持 success / error / warning / info。
- [ ] 检查 `copy.js` 是否包含复制失败降级。
- [ ] 检查 `debounce.js` 是否能用于输入防抖。
- [ ] 检查 `storage.js` 是否适合历史记录和设置保存。
- [ ] 如确有需要，再新增 `loading.js`。
- [ ] 如确有需要，再新增 `themes.css`。

### 验收

- [ ] 工具函数可直接通过 `<script>` 引入。
- [ ] 无需 npm、构建或打包。
- [ ] 任意工具页可独立使用这些函数。

---

## 四、任务 2：统一复制体验

### 目标

所有复制操作使用统一复制函数，并提供一致提示。

### 操作

- [ ] 搜索各页面中的复制逻辑。
- [ ] 选择 `password-generator.html` 或 `key-generator.html` 作为试点。
- [ ] 引入 `js/utils/toast.js`。
- [ ] 引入 `js/utils/copy.js`。
- [ ] 将原有复制逻辑替换为 `copyToClipboard()`。
- [ ] 验证复制成功提示。
- [ ] 验证复制失败提示。
- [ ] 推广到其他工具页。

### 验收

- [ ] 复制成功有统一 Toast。
- [ ] 复制失败有明确提示。
- [ ] 不再维护多份重复复制代码。

---

## 五、任务 3：统一常规反馈

### 目标

用非阻塞 Toast 替代常规 `alert`。

### 操作

- [ ] 搜索所有 `alert(`。
- [ ] 判断哪些是常规反馈，哪些是必要阻塞确认。
- [ ] 常规反馈替换为 Toast。
- [ ] 错误场景使用 `Toast.error()`。
- [ ] 成功场景使用 `Toast.success()`。
- [ ] 警告场景使用 `Toast.warning()`。

### 验收

- [ ] 常规操作不再弹出阻塞式 alert。
- [ ] 错误信息仍然清楚。
- [ ] Toast 不遮挡主要按钮。

## 九、任务 7：性能优化

### 目标

只优化真实会卡顿的场景。

### 优先处理

#### 字数统计

- [ ] 大文本输入时增加 debounce。
- [ ] 大文件解析时显示 Loading。
- [ ] 如果仍卡顿，再考虑 Web Worker。

#### YAML 工具

- [ ] 格式化/对比大文本时增加 Loading。
- [ ] 输入解析增加 debounce。
- [ ] 错误定位保持清晰。

#### 翻译工具

- [ ] 输入触发请求时增加 debounce。
- [ ] 请求中显示 Loading。
- [ ] 请求失败显示明确错误。

#### 剪贴板历史

- [ ] 历史列表过长时限制渲染数量。
- [ ] 搜索使用 debounce。
- [ ] 必要时迁移到 IndexedDB。

### 验收

- [ ] 大输入时页面不卡死。
- [ ] 用户知道当前是否正在处理。
- [ ] 优化不增加使用门槛。

## 十三、每次提交前检查

```bash
git status
```

手动检查：

- [ ] 首页可打开。
- [ ] 修改过的工具页可打开。
- [ ] 主要功能可用。
- [ ] 控制台无明显错误。
- [ ] 移动端基本可用。
- [ ] 没有新增构建依赖。
- [ ] 没有新增后端依赖。

推荐提交粒度：

```bash
git add <files>
git commit -m "docs: simplify lightweight upgrade checklist"
git commit -m "refactor: use shared copy utility in password generator"
git commit -m "feat: add toast feedback to key generator"
```

---

## 十四、执行原则总结

- 不追求一次性重构完成。
- 不为了架构而架构。
- 不破坏静态直开能力。
- 优先解决真实体验问题。
- 每次只做一类清晰改动。
- 每个工具始终保持轻量、独立、可下载使用。
