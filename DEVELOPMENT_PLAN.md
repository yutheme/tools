# 工具集升级步骤清单

**项目**：`yutheme/tools`  
**定位**：可直接下载使用的轻量级静态工具集合  
**升级方式**：小步增强，不做重型重构

---

## 一、升级目标

本项目升级的目标不是把工具集改造成复杂 Web App，而是在保持轻量、静态、可直接打开使用的前提下，逐步提升：

- 使用体验
- 视觉一致性
- 移动端适配
- 本地存储可靠性
- 大文件处理性能
- 工具可维护性

---

## 二、必须遵守的原则

### 1. 保持静态直开

任何升级都不能破坏以下能力：

- 下载项目后可直接打开 `index.html`。
- 单个工具页面可直接打开使用。
- 不需要后端服务。
- 不需要构建命令。
- 不依赖路由服务。

### 2. 不引入重型依赖

默认不引入：

- React / Vue / Angular 等框架。
- Vite / Webpack 等构建链。
- Node 后端服务。
- 复杂模板系统。

如果确实需要第三方库，应满足：

- 可以通过普通 `<script>` 或 `<link>` 引入。
- 不影响核心功能的本地使用。
- 有清晰的降级方案。

### 3. 先试点，再推广

每类升级先选一个工具页试点，验证后再推广：

1. 选一个简单页面。
2. 做最小改动。
3. 本地打开验证。
4. 再推广到其他页面。

### 4. 优先修复真实问题

不为“架构完整”而重构。优先处理：

- 重复代码明显影响维护。
- 复制、提示、错误反馈体验不好。
- 移动端显示不佳。
- 大文本/大文件导致卡顿。
- 颜色和样式不统一。

---

---

## 四、步骤 1：统一基础交互反馈

### 目标

用轻量 Toast 替代页面中的 `alert`，让成功、失败、复制、加载等反馈更自然。

### 涉及文件

- 已有：`js/utils/toast.js`
- 逐步修改：各工具 HTML 页面

### 执行方式

1. 检查 `js/utils/toast.js` 是否能直接通过 `<script>` 引入。
2. 在一个工具页试点引入 Toast。
3. 将复制成功、错误提示从 `alert` 改为 `Toast.success()` / `Toast.error()`。
4. 检查移动端提示位置是否合理。
5. 推广到其他工具页。

### 验收标准

- [ ] 不再使用阻塞式 `alert` 作为常规反馈。
- [ ] 成功、失败、警告提示样式清晰。
- [ ] 多条提示不会遮挡主要操作。
- [ ] 本地直接打开页面时 Toast 正常工作。

---

## 五、步骤 2：统一复制能力

### 目标

所有复制操作使用同一套复制函数，减少重复代码，并提供失败降级。

### 涉及文件

- 已有：`js/utils/copy.js`
- 逐步修改：
  - `password-generator.html`
  - `key-generator.html`
  - `yaml-previewer.html`
  - `word-count.html`
  - `number-converter.html`
  - `clipboard-history.html`

### 执行方式

1. 盘点所有复制按钮和复制函数。
2. 选一个工具页替换为 `copyToClipboard()`。
3. 确认复制成功时有 Toast 提示。
4. 确认复制失败时有可理解的提示。
5. 分批推广。

### 验收标准

- [ ] 复制成功有统一提示。
- [ ] 复制失败不静默。
- [ ] 不再重复维护多份复制逻辑。
- [ ] 支持现代浏览器的 Clipboard API。

---

## 六、步骤 3：完善共享工具函数

### 目标

让多个工具都能复用基础能力，但不强迫所有逻辑模块化。

### 当前已有

- `js/utils/debounce.js`
- `js/utils/storage.js`
- `js/utils/toast.js`
- `js/utils/copy.js`

### 可补充

- `js/utils/loading.js`：轻量 Loading。
- `css/themes.css`：主题变量。

### 执行方式

1. 只补充确实会被多个页面使用的工具函数。
2. 每个工具函数保持独立，无构建依赖。
3. 每个工具函数暴露到 `window`，方便 HTML 直接使用。
4. 在 README 或注释中写明简单用法。

### 验收标准

- [ ] 工具函数可以直接通过 `<script>` 引入。
- [ ] 不依赖打包工具。
- [ ] 不污染过多全局变量。
- [ ] 命名清晰，容易理解。

---

## 七、步骤 4：逐个工具页接入共享能力

### 目标

在不改变页面基本结构的前提下，逐个提升工具页体验。

### 推荐顺序

1. `password-generator.html`
2. `key-generator.html`
3. `number-converter.html`
4. `word-count.html`
5. `yaml-previewer.html`
6. `translator.html`
7. `clipboard-history.html`
8. `vrime.html`

### 每个页面的处理清单

- [ ] 引入需要的 `js/utils/*`。
- [ ] 替换重复复制逻辑。
- [ ] 替换常规 `alert`。
- [ ] 高频输入增加 debounce。
- [ ] 检查错误提示。
- [ ] 检查移动端布局。
- [ ] 本地直接打开验证。

---

## 八、步骤 5：优化移动端体验

### 目标

让工具在手机浏览器中也能清楚、可点、可用。

### 优先检查

- 输入框宽度。
- 按钮大小。
- 表单间距。
- 结果区域换行。
- 页面横向滚动。
- Toast 是否遮挡操作。

### 验收标准

- [ ] 375px 宽度下可正常使用。
- [ ] 按钮点击区域足够大。
- [ ] 输入框不会溢出屏幕。
- [ ] 结果文本可复制、可阅读。

---

## 九、步骤 6：增加轻量主题能力

### 目标

通过 CSS 变量逐步实现更统一的颜色系统，并为深色模式做准备。

### 涉及文件

- 可新增：`css/themes.css`
- 修改：`css/shared.css` 和各工具页面内联样式

### 执行方式

1. 提取最常用的颜色变量。
2. 先在一个工具页试点替换硬编码颜色。
3. 检查浅色模式显示是否不变。
4. 再增加深色变量。
5. 最后逐页替换。

### 验收标准

- [ ] 浅色模式不退化。
- [ ] 深色模式文字可读。
- [ ] 不因主题逻辑影响工具功能。
- [ ] 用户选择可保存在本地。

---

## 十、步骤 7：优化性能敏感场景

### 目标

只针对真正卡顿的场景做性能优化。

### 优先场景

- 字数统计处理大文本。
- YAML 格式化/对比处理大文件。
- 翻译工具高频输入触发请求。
- 剪贴板历史列表过长。

### 可选手段

- debounce。
- 分批渲染。
- Loading 状态。
- Web Worker。
- IndexedDB。

### 验收标准

- [ ] 大输入不会明显卡住页面。
- [ ] 异步操作有可见反馈。
- [ ] 优化没有显著增加使用复杂度。

## 十一、步骤 8：按需新增工具

### 候选工具

- Markdown 编辑器。
- JSON 格式化/校验工具。
- 二维码工具。
- 颜色工具。
- 正则测试器。

- 单独 HTML 可运行。
- 无后端依赖。
- 能从首页进入。

## 十三、每次改动的通用验收

每次提交前检查：

- [ ] `index.html` 能打开。
- [ ] 修改过的工具页能打开。
- [ ] 主要功能正常。
- [ ] 控制台无明显错误。
- [ ] 移动端基本可用。
- [ ] 不需要构建步骤。
- [ ] 不需要后端服务。
- [ ] 隐私数据仍在本地处理。

---

## 十四、推荐提交方式

每次只提交一个清晰主题：

```bash
git add <changed-files>
git commit -m "feat: add shared toast feedback"
git commit -m "refactor: use shared copy utility in password generator"
git commit -m "style: improve mobile layout for yaml tool"
git commit -m "docs: simplify upgrade plan for lightweight static tools"
```

---

## 十五、总结

升级不是一次性大重构，而是持续小步优化。

正确方向是：

- 保持轻量。
- 保持静态。
- 保持可直接下载使用。
- 先解决真实体验问题。
- 逐个工具稳定推进。
